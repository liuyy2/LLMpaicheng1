## 一、论文定位

### 1.1 研究对象
本文研究对象为**高频扰动条件下火箭发射场多任务共享稀缺资源的动态调度问题**。其本质可归结为带时间窗口、共享资源竞争和动态扰动的资源约束项目调度问题。

### 1.2 核心矛盾
发射场调度并非单纯追求低延迟，而是在**准时性**与**计划稳定性**之间进行权衡：
- 若每次扰动后都进行全局重排，虽然可能降低局部延迟，但会显著增加计划漂移与执行代价；
- 若过度强调计划不变，则难以及时响应窗口收缩、设备故障和工序延误等突发事件。

### 1.3 文章定位
本文宜定位为**方法型、工程导向型动态调度论文**，而不是“系统实现论文”或“LLM 应用展示论文”。

全文必须围绕以下主线展开：

**高密度发射任务背景 -> 发射场动态调度问题 -> 高频扰动与稳定性矛盾 -> 滚动重调框架 -> 根因驱动局部修复机制 -> 对比实验验证 -> 适用范围与局限**

### 1.4 题目建议
**主推题目：**
- 面向高频扰动的火箭发射场滚动时域动态调度与根因驱动局部修复方法

**备选题目：**
- 考虑计划稳定性的火箭发射场滚动重调优化方法
- 基于时序资源冲突诊断与局部修复的火箭发射场动态调度方法
- 火箭发射场多任务共享资源场景下的动态重调与局部修复方法

> 不建议把“大语言模型”直接写入标题。你的真正辨识度不是“用了 LLM”，而是“根因驱动局部修复是否改善了动态重调质量”。

## 二、论文研究摘要（推荐作为后续摘要母稿）

### 2.1 研究摘要长版
针对高频扰动条件下火箭发射场多任务共享稀缺资源导致的动态重调问题，构建考虑发射窗口约束、共享资源竞争、冻结机制和扰动触发重调的动态调度模型，以加权延迟最小和计划漂移最小为优化目标。为兼顾求解效率与执行稳定性，提出一种根因驱动的局部修复方法：首先利用时序资源冲突图识别扰动后计划中的关键冲突与根因任务；然后在结构化诊断摘要基础上，由大语言模型生成小规模解锁任务集；最后结合 Anchor Fix-and-Optimize 机制实施局部重排，并通过分级回退策略增强复杂扰动下的可行解恢复能力。基于含窗口收缩、设备故障和工序延误等扰动的滚动仿真场景开展对比实验，从准时性、稳定性、局部性和求解效率等维度对所提方法进行验证。结果表明，所提方法能够在保证准时性的同时降低计划漂移和不必要的全局调整，验证了其在发射场动态调度场景中的有效性与鲁棒性。

### 2.2 期刊摘要压缩版（可直接继续打磨）
针对高频扰动下火箭发射场多任务共享稀缺资源引起的动态调度问题，构建考虑发射窗口、资源竞争和冻结机制的滚动动态调度模型，以加权延迟最小和计划漂移最小为目标。提出一种根因驱动的局部修复方法，利用时序资源冲突图识别关键冲突，在结构化诊断摘要基础上由大语言模型生成解锁任务集，并结合 Anchor Fix-and-Optimize 机制实施局部重排，同时设计分级回退策略提升复杂扰动下的可行解恢复能力。通过多扰动场景下的滚动仿真实验进行验证。结果表明，所提方法能够在保证准时性的同时改善方案稳定性，减少不必要的全局调整，具有较好的有效性与鲁棒性。

### 2.3 摘要写作硬约束
1. 摘要必须出现四个关键词：**高频扰动、滚动重调、计划稳定性、根因驱动局部修复**。

4. 摘要结果表述要克制，优先使用“能够有效降低……”“在保证……的同时改善了……”“验证了有效性与鲁棒性”。

## 三、研究问题、意义与论文切入点

### 3.1 研究问题
如何在天气窗口收缩、设备故障、工序延误等高频扰动条件下，对多任务共享稀缺资源的火箭发射场计划实施高质量的在线动态重调，并在保证可行性的前提下兼顾任务准时性与方案稳定性？

### 3.2 工程意义
发射场任务组织具有窗口严格、资源耦合强、工序链条长和执行代价高等特点，一旦在扰动响应中进行不必要的大范围调整，容易引发任务链式延误、窗口丢失和现场保障压力上升。因此，研究能够快速响应扰动且尽量保持方案稳定的动态重调方法，具有明确的工程应用价值。

### 3.3 学术意义
现有发射场研究更多集中于静态并行规划、流程优化与智能化系统建设，对于“高频扰动 + 稳定性约束 + 根因驱动局部修复”这一动态调度问题的研究仍相对不足。本文试图将滚动重调、稳定性建模与根因驱动局部修复机制有机结合，为发射场动态调度研究提供一种更符合实际执行逻辑的求解思路。

### 3.4 切入角度
本文最合适的切入角度不是“LLM 很强”，而是：

**在高频扰动下，发射场调度不能每次都全局推倒重来；真正有价值的是识别造成失稳的关键冲突，并在保证可行性的前提下尽量局部修复。**

## 四、文献定位与本文缺口（供引言直接复用）

### 4.1 相关研究的三条主线
1. **发射场多任务并行规划研究**：重点解决任务流程梳理、资源占用表达与并行规划求解问题，为发射场调度建模提供了工程基础。
2. **流程优化与智慧发射场研究**：强调流程数字化、测发保障能力提升和高密度发射组织需求，说明发射场运行已呈现更强的动态性与复杂性。
3. **滚动时域动态调度研究**：在其他航天任务场景中，滚动时域、事件触发重调和“性能-调整代价”并重的建模方式已经形成较成熟写法，可作为本文的重要方法母本。

### 4.2 本文要明确指出的缺口
后续引言不要泛泛写“现有研究不足”，而应明确写成：

- 现有发射场并行规划研究主要面向**静态计划生成**，对扰动驱动的在线动态重调关注不足；
- 现有工程应用与流程优化研究强调保障能力提升，但对**计划稳定性约束下的数学建模与优化求解**展开不够；
- 现有滚动优化研究虽然提供了动态调度范式，但在发射场场景下，对**根因识别、局部修复范围控制及可行性回退机制**的研究仍不足。

### 4.3 引言中可直接使用的过渡句
- 随着高密度发射需求持续增长，发射场任务组织呈现并行化、动态化和高约束化特征。
- 与静态规划问题不同，发射场动态调度不仅需要响应突发扰动，还需控制方案调整幅度，降低对现场执行的冲击。
- 因此，如何在复杂约束下实现兼顾准时性与稳定性的动态重调，成为当前亟待解决的关键问题。

## 六、统一的问题建模口径

### 6.1 场景抽象
- **任务层**：每个发射任务由若干串行工序组成，末端关键工序需落入有效发射窗口；
- **资源层**：发射台、测试设备、联测资源等为共享稀缺资源；
- **时间层**：同时存在任务自身时间窗、全局开放窗口和工序前后序约束；
- **扰动层**：包括窗口收缩、设备故障、工序延误等；
- **执行层**：已开始工序及冻结视野内工序在本轮重调中保持不变。

### 6.2 决策目标
建议全文始终坚持“两阶段词典序优化”的统一口径：

- **阶段 1：准时性优先**，以加权延迟最小为目标；
- **阶段 2：稳定性优先**，在延迟相对最优值受控放宽的条件下，以计划漂移最小为目标。

### 6.3 计划稳定性定义
计划稳定性并非“尽量不动”这么简单，而是对重调引起的**关键工序时间偏移、时间窗切换和序列变化**进行综合刻画。后续写作时，Drift 的定义必须服务于一句话：

**Drift 用于衡量策略造成的、原本可以避免的计划变动代价。**

### 6.4 约束条件
后续模型章节建议固定写为：
1. 工序前后序约束；
2. 发射窗口与全局开放窗口约束；
3. 共享资源容量约束；
4. 工艺最大等待时间约束；
5. 冻结视野约束；
6. 局部修复时的锚点约束。

### 6.5 可直接放入正文的模型说明句
- 在分析发射场任务流程与资源占用特性的基础上，构建了考虑发射窗口、共享资源竞争和冻结机制的动态调度模型。
- 为刻画重调引起的方案调整代价，定义计划漂移指标，用以衡量关键工序时间偏移及序列变化带来的执行扰动。
- 为兼顾准时性与稳定性，采用两阶段词典序优化方式求解动态调度子问题。

## 七、统一的方法主线

### 7.1 滚动重调框架
采用滚动时域策略处理动态调度问题，将长周期动态决策过程划分为若干连续的重调区间；在每个区间内，根据最新扰动状态对未来视野内未完成任务进行重排。

### 7.2 根因诊断
当扰动发生后，首先从当前计划中识别共享资源上的关键冲突，构建时序资源冲突图，对冲突簇、瓶颈资源压力和任务紧急度进行综合分析，定位导致计划失稳的根因任务。

### 7.3 局部修复
在结构化冲突诊断结果基础上，由大语言模型辅助生成需要解锁的任务集合。解锁任务允许在本轮求解中重新安排，未解锁任务保持锚定，从而将“大范围全局重排”转化为“面向关键冲突的小范围局部修复”。

### 7.4 精确求解与回退机制
对解锁任务采用 Anchor Fix-and-Optimize 方式实施局部重排；若局部修复失败，则通过逐步扩大解锁范围、缩短冻结视野、放宽稳定性容差等方式进行分级回退，并在必要时退化为全局重排，以保证最终可行性。

### 7.5 方法章节必须强调的三句话
- 本文并非简单地在每次扰动后进行全局重排，而是先识别关键冲突根因，再决定局部修复范围。
- 本文的重点不是让大语言模型直接替代求解器，而是利用其对结构化冲突摘要进行辅助决策，缩小后续精确优化的搜索范围。
- 回退链不是工程细节，而是保证复杂扰动场景下方法可行性与鲁棒性的必要组成部分。

### 7.6 建议的算法流程表述
后续正文算法部分建议写成以下 6 步：
1. 采集当前滚动窗口内的任务、资源与扰动状态；
2. 构建时序资源冲突图并识别根因任务；
3. 生成结构化诊断摘要；
4. 输出解锁任务集并确定局部修复范围；
5. 通过两阶段优化完成局部重排；
6. 若求解失败，触发分级回退直至得到可行方案。

## 八、实验设计母稿（面向论文正文）

### 8.1 实验目的必须回答的四个问题
1. 所提方法能否在高频扰动下兼顾准时性与计划稳定性？
2. 根因驱动局部修复是否能够减少不必要的全局调整？
3. 与非 LLM 的局部修复方法相比，结构化诊断 + LLM 辅助决策是否具有增益？
4. 在复杂扰动下，回退机制是否能够保证可行方案输出？

### 8.2 建议的对比基线分层
**主比较组（建议放主文表格）：**
- FixedWeightPolicy：固定参数全局重排基线；
- NoFreezePolicy 或 MinimalFreezePolicy：突出冻结机制的作用；
- GARepairPolicy：非 LLM 局部修复基线；
- 本方法：TRCG + LLM 辅助 + Anchor Fix-and-Optimize。

**补充比较组（建议放附录或扩展分析）：**
- RealLLMPolicy：用于说明“从元参数调整到局部修复”的方法演进；
- GreedyPolicy：作为弱基线说明工程启发式方法的局限。

### 8.3 指标体系推荐口径
**主指标：**
- `weighted_tardiness`：衡量任务准时性；
- `drift_per_replan`：衡量单次重调平均扰动代价；
- `episode_drift`：衡量全局稳定性；
- `feasible_rate`：衡量复杂扰动下的方案可行性；
- `avg_solve_time_ms`：衡量求解效率。

**机制解释指标：**
- `unlock_size_avg`：衡量修复局部性；
- `fallback_rate`：衡量回退机制触发频率；
- `total_switches`：衡量序列扰动强度。

**补充指标：**
- `avg_delay`、`on_time_rate`、`completion_rate`。

> 主文不建议把评价体系做得过于复杂。只要把“准时性-稳定性-局部性-鲁棒性”四条线讲清楚，就足够支撑发文要求。

- “最小任务集” -> “小规模解锁任务集”
